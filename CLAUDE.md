## Project Overview

SkywalkerRoaster is an Arduino-based open-source project that enables control and monitoring of the **Skywalker/ITOP MTCR (SkItop) coffee roaster** via the **Artisan Scope** application. The Arduino microcontroller sits between the roaster's controller and Artisan, intercepting the proprietary serial protocol to expose temperature readings and accept control commands.

## Sketch Roles

### SkyCommand (`SkyCommand/SkyCommand.ino`)
The primary sketch — use this for actual roaster control.
- Emulates the TC4 protocol so Artisan treats the Arduino as a known device
- Reads temperature from roaster messages (7-byte proprietary protocol)
- Forwards control commands from Artisan to the roaster (heat/vent/drum/fan/cool)
- **Safety features:** 10-second command timeout, 300°C over-temperature emergency stop, checksum validation

### SkywalkerSpy (`SkywalkerSpy/SkywalkerSpy.ino`)
Logging-only sketch — use this to observe the roaster without controlling it.
- Monitors both Tx lines (controller and roaster) passively
- Outputs `TEMP,HEAT_DUTY,VENT_DUTY` lines for Artisan logging
- No write capabilities; useful for protocol analysis and debugging

### SkyCommandPerfBeta (`SkyCommandPerfBeta/SkyCommandPerfBeta.ino`)
Work-in-progress performance-optimized variant of SkyCommand. Not yet stable for production use.

## Roaster Protocol

The roaster uses a custom timing-based serial protocol (not standard UART).

**Controller → Roaster (6 bytes):** Fan, Filter, Cool, Drum, Heat duty cycles, Checksum  
**Roaster → Controller (7 bytes):** ADC value A (bytes 0–1), ADC value B (bytes 2–3), 2 mystery bytes, Checksum

Key timing constants:
- Preamble: ~7.5 ms pulse
- Bit threshold: 1200 µs (above = binary `1`, below = binary `0`)
- Messages repeat every ~10 ms
- Transmission order: LSB first

**Checksum:** Simple byte sum of all preceding bytes (no overflow protection).

### Temperature Calculation
Temperature is derived from ADC values A and B via a 4th-degree polynomial regression model:

```
T = f(A, B)  →  16 coefficients embedded in sketch
```

The coefficients are generated by `Data/model3.py` using `scikit-learn` on the labeled dataset in `Data/RealLabeledTemperatures.txt`.

## Artisan Integration

Artisan communicates with the Arduino via the **TC4 emulation protocol** over USB serial. Supported commands handled by SkyCommand:

| Command  | Action                                      |
|----------|---------------------------------------------|
| `READ`   | Return current temperatures (T1, T2)        |
| `OT1`    | Set heat duty cycle (0–100)                 |
| `OT2`    | Set vent duty cycle (0–100)                 |
| `DRUM`   | Set drum speed                              |
| `FILTER` | Toggle filter                               |
| `COOL`   | Activate cooling                            |
| `OFF`    | Turn off                                    |
| `ESTOP`  | Emergency stop                              |
| `CHAN`   | Channel configuration                       |
| `UNITS`  | Set temperature units (C/F)                 |

Artisan configuration presets (`.aset` files) are in `ArtisanConfigs/`. Import the appropriate file for your setup via Artisan's config import feature.

## Development Workflow

### Building / Compiling
There is no Makefile or build script. Compile sketches using the **Arduino IDE** directly.

To verify compilation without hardware, the CI/CD pipeline uses `arduino/compile-sketches@v1`.

### CI/CD (GitHub Actions)
File: `.github/workflows/compile-sketch.yml`

- **Trigger:** Every push and pull request
- **Boards tested:** `arduino:avr:nano`, `arduino:avr:micro`, `arduino:avr:leonardo`
- **Sketches compiled:** `SkyCommand/`, `SkywalkerSpy/`

When modifying sketches, ensure CI passes on all three board targets.

### Python (Data / model3.py)
Dependencies: `scikit-learn`, `numpy`, `matplotlib`

```bash
# Train the temperature model and print coefficients
python Data/model3.py
```

Paste the output coefficients into the `TEMP_COEFFICIENTS` array in `SkyCommand.ino` when the model changes.

## Code Conventions

### Arduino Sketches
- **Debug macros:** `#define __DEBUG__` and `#define __WARN__` gate verbose logging — keep these at the top of each sketch
- **Pin constants:** Defined at top of file as named constants (e.g., `txPin`, `rxPin`)
- **Timing constants:** All in microseconds; add inline comments with human-readable equivalents
- **Buffers:** Fixed-size `uint8_t` arrays for protocol messages
- **Timeout constants:** Prefixed with `TIMEOUT_` (e.g., `TIMEOUT_PREAMBLE_SEARCH`)
- **Failsafe pattern:** Use boolean flags (`failedToReadRoaster`) and retry counters (`roasterReadAttempts`) rather than exceptions
- **Safety limits:** Always validate temperature against `MAX_TEMP` (300°C) before forwarding commands

### Python
- Use `scikit-learn` for model training
- Use `numpy` for numerical operations
- Use `matplotlib` for diagnostic plots
- Scripts are intentionally simple utilities — no abstraction layers needed

## Safety Notes

- **Never disable the 10-second timeout** — it is the primary failsafe if USB communication drops
- **Never raise `MAX_TEMP` above 300°C** — exceeding this risks hardware damage
- **Always validate checksums** before acting on a roaster message
- The roaster has no acknowledgment mechanism; commands are fire-and-forget
